# `work_summary` — period (monthly) work summaries

`work_summary` summarizes a time period's work — typically a calendar month —
across the git repositories in a workspace, as bullet-pointed Markdown / Org / JSON.
It formalizes the manual workflow of "summarize what I did in May across
`~/Workspace/<org>`" into a single MCP tool call.

- **Tool body:** `src/mcp/tools/tool_work_summary.rs`
- **Engine:** `src/worklog/` (`mod` · `repos` · `git` · `enrich` · `narrative` · `report`)
- **Params:** `src/mcp/params/worklog.rs` (`WorkSummaryParams`)
- **Config:** `[worklog]` in `src/config.rs` (`WorklogConfig`)
- **Formal slice:** `docs/formal/tla/WorkSummaryBoundary.tla` + `docs/formal/work-summary-traceability.md`

## Authority model

**Live git is the authoritative source for every fact; the temporal-graph index is
a freshness-gated enrichment, never a correctness dependency.** Rationale (grounded
in the schema):

- Per-commit **line churn is not stored** in `git_commits` (`module_growth` emits
  `additions`/`deletions` as `NULL`), so churn must come from a live
  `git log --numstat` pass.
- **Uncommitted / mid-stream state and live file mtimes are never in the DB** — they
  are working-tree state, read live (`git status` / `git diff`, via
  `src/deps/gitstate.rs`).
- The unified/temporal graph is per-project `[git] index_history`-opt-in and
  cron-populated, so it can lag or be empty.

The tool therefore reads commit facts live and uses the DB only for (a) **canonical
project enumeration** and (b) **optional enrichment** behind a freshness gate (below).

## Parameters

| Param | Type | Default | Notes |
|-------|------|---------|-------|
| `workspace_root` | string | first `[workspace]` path | Absolute or `~`-prefixed. Repos under this path are summarized. |
| `month` | `"YYYY-MM"` | current UTC month | Whole calendar month. Ignored when `since`+`until` are both given. |
| `since` / `until` | `"YYYY-MM-DD"` or RFC3339 | — | Explicit half-open window `[since, until)`. `since >= until` is rejected. |
| `author` | string | local `git config user.name` | A case-insensitive `git --author` regex ("my work"). `"all"` = every contributor. |
| `format` | `markdown\|org\|json` | `markdown` | Other `ReportFormat` values are rejected (this tool renders three). |
| `group_by` | `project\|theme\|week` | `project` | Adds a primary rollup; the per-project breakdown is always included. |
| `include_uncommitted` | bool | `true` | Include the mid-stream working-tree section. |
| `use_graph` | `auto\|on\|off` | `auto` | Temporal-graph enrichment mode (see freshness gate). |
| `narrative` | bool | `false` | LLM prose bullets per project (see Narrative). |
| `max_repos` | int | 200 (clamp 1..=1000) | Cap on repos scanned. |
| `limit` | int | 100 (clamp 1..=1000) | Cap on projects rendered (busiest first). |

All inputs are validated / normalized / clamped at the boundary
(`WorkSummaryRequest::from_params`) **before any repo scan or DB query**; the resolved
parameters are echoed back in the report's `normalized` block.

## Configuration — `[worklog]`

Per-call params always override these; the section just sets the defaults.

```toml
[worklog]
default_format       = "markdown"   # markdown|org|json
default_author       = ""           # empty → resolve local git user; "all" for everyone
graph_enrichment     = "auto"       # auto|on|off
narrative_default    = false
max_repos            = 200
max_projects         = 100
narrative_backend    = "qwen3-4b"   # qwen3-4b (~2.5 GB) | qwen3-8b (~5 GB) | other → deterministic
narrative_max_tokens = 160          # token budget per project
```

## Output

Every rendition leads with a headline (period · author · projects · commits · active
days · churn) and a Unicode **cadence sparkline** (`render::glyphs::SPARK`) over the
per-day commit counts, plus the conventional-commit **type mix**. Then:

- `group_by=week` → a **By week** section (commits / churn / distinct projects per ISO week)
- `group_by=theme` → a **By theme** section (commits per conventional-commit type)
- always → a **By project** section: one header bullet per project
  (`N commits · +A/−B · first→last`) with sub-bullets for top scopes/keywords,
  representative commit subjects, the uncommitted/mid-stream note, optional narrative
  bullets, and the enrichment/freshness line.

`format=json` returns the structured `WorkSummaryReport` (its `normalized` block
echoes the resolved params, including `narrative_engine`).

## Narrative (`narrative=true`)

When enabled, each project's bullets are generated by a **local qwen3** model
(`narrative_backend`, default `qwen3-4b`) from the collected facts. The model is
loaded once from the HuggingFace cache (downloaded on first use, reused thereafter;
shared with the memory subsystem's qwen3 cache) and run inside `tokio::task::spawn_blocking`,
so the multi-GB load + greedy decode never stall the async daemon. On **any** failure
— no GPU, model absent, unknown backend, generation error — each project falls back to
the **deterministic `compose` prose** (fully reproducible). The engine that actually
ran is surfaced in `normalized.narrative_engine` (`"qwen3-4b"` or a
`"deterministic…"` reason). Off by default to keep the artifact reproducible (it can
back an invoice).

## Worktree canonicalization

Repos are deduped to one canonical project per worktree group, reusing
`db::queries::pick_main_worktree_ids` — group by `projects.git_common_dir` (worktrees
of one repo) else `git_root_commits` (sibling clones), keeping the shortest basename.
Genuinely distinct projects (e.g. `f1r3node` vs `f1r3node-rust`, different root
commits) stay separate. Because `git log --all` runs on the canonical worktree's
shared object store, feature-branch worktrees never double-count.

## Enrichment & the freshness gate

For each project, `src/worklog/enrich.rs` computes (best-effort):

- **freshness** — live `git rev-parse HEAD` vs the indexed `git_last_commit:<id>`
  watermark → `fresh` | `stale` | `unindexed`.
- **reconciliation** — indexed `git_commits` count in the window vs the live count →
  `match` | `index_behind` | `index_ahead` | `n/a`.
- **topics** — top indexed `code_topics` labels for the project, attached **only**
  when `use_graph != off` AND freshness is `fresh`.

The live-git numbers are always authoritative; enrichment never overrides them, and
any downgrade is recorded in the report.

## Known limitations

- **Author-date vs committer-date.** `git log --since/--until` filters by *committer*
  date, while the display (`%ad`/`--date=short`) and the enrichment reconciliation
  (`git_commits.author_date`) use *author* date. For rebased / cherry-picked commits
  these differ, so the histogram bucket and the reconciliation count can be slightly
  off at window edges. `git_commits` has no committer-date column to reconcile against;
  reconciliation is therefore an approximate signal by design.
- **Churn is live-only.** Sourced from `git log --numstat` each call; not cached. (A
  future enhancement could backfill `git_commit_files` with numstat columns.)
- **`render` is hand-rolled** (Markdown/Org/JSON) reusing only `render::glyphs`;
  `latex`/`html`/`text` are intentionally rejected.

## Module layout (reconstruct-from-scratch)

| File | Responsibility |
|------|----------------|
| `src/worklog/mod.rs` | `WorkSummaryRequest::from_params` (validate/normalize/clamp), `summarize()` orchestration, the `WorkSummaryReport` model, cross-cutting rollups (`build_totals`/`build_weeks`/`build_themes`). |
| `src/worklog/repos.rs` | `canonical_repos()` — enumerate + worktree-dedup + filter under `workspace_root`. |
| `src/worklog/git.rs` | live `git log --numstat` collection, conventional-commit type/scope parsing, daily histogram, `git status`/`diff` uncommitted state. |
| `src/worklog/enrich.rs` | freshness gate + reconciliation + topic enrichment. |
| `src/worklog/narrative.rs` | qwen3 generation (`spawn_blocking`) + deterministic `compose` fallback. |
| `src/worklog/report.rs` | `to_markdown` / `to_org` / `to_json`. |

Registration: a `#[tool]` handler in `src/mcp/server/handlers/inventory.rs`
(`router_inventory`) + a `dispatch_tool!` arm `"work_summary" => work_summary(WorkSummaryParams)`
in `src/mcp/server.rs`. The module is declared in both `src/lib.rs` and `src/main.rs`
(the bin re-declares the module tree).

## Formal verification

`docs/formal/tla/WorkSummaryBoundary.tla` models the request boundary: all rejects are
local (zero scans/queries/renders), the repo scan is bounded by
`min(available, clamp(max_repos))`, per-repo reads are bounded (one git pass + one
reconciliation query), enrichment is freshness-gated, and accepted calls render one
canonical-format envelope. Capped-TLC: 11 states, 8 invariants, 0 violations
(`scripts/tlc-capped.sh WorkSummaryBoundary.tla`). See
`docs/formal/work-summary-traceability.md`.

## Testing

- **Unit** (`src/worklog/`): conventional-commit parsing, numstat/shortstat parsing,
  month/instant parsing, top-n ordering, path-under/dir-normalization, `parse_bullets`
  marker-stripping, `variant_of`.
- **Oracle** (`pgmcp-testing/tests/oracle_work_summary.rs`): a deterministic real-DB
  test that seeds a throwaway git repo with three known commits and asserts the exact
  commit count / churn / type mix / span / normalized-echo / bad-format rejection
  end-to-end through `call_tool_cli`.
- The full-workspace "reproduce the May figures (≈1,402 commits / 19 projects /
  29 active days)" check is a live post-restart verification (the index is mutable, so
  it isn't a fixture assertion).

## Activation

`work_summary` is exposed by the warm MCP daemon only after a release rebuild +
daemon restart (the standard pgmcp activation pattern). Example call:

```
work_summary { workspace_root: "~/Workspace/f1r3fly.io", month: "2026-05", author: "dylon" }
```
