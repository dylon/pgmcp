## 1. Problem statement

Reproduction:
```sh
./target/release/pgmcp daemon &
PGMCP_DAEMON_PID=$!
tail -f ~/.local/share/pgmcp/pgmcp.log
kill $PGMCP_DAEMON_PID
```
**Symptom:** RSS climbs unboundedly until the OOM-killer terminates the daemon. Observed on 128 GiB hardware, so the leak/accumulation reaches ≥ tens of GiB.

**User hypothesis (confirmed):** the recent 13,120-line addition (git diff --stat HEAD) introducing topic-clustering (`src/cron/topic_clustering.rs`, 1226 lines), graph analysis (`src/cron/graph_analysis.rs`, 543 lines + `src/graph/*`), similarity scan (`src/cron/similarity.rs`, 132 lines), daemon lifecycle (`src/daemon_state.rs`, 262 lines), and MCP server expansion (5719 lines) introduced the regression.

## 2. Root cause — four compounding problems

**P1.** Global topic clustering (`run_global_topic_scan`) loads *all* chunks across *all* projects into anonymous heap:
- `bulk_extract_embeddings(pool, None)` → `Vec<ChunkEmbeddingRow>` ≈ 1–3 GB for 113k chunks.
- `Array2<f64>` data (347 MB) + redundant `Vec<Vec<f64>>` `data_vecs` duplicate (347 MB).
- FCM per-iteration allocation churn: `membership.clone()` + `u_pow_m = membership.mapv(...)` + `dist_sq` + `dot_xc` ≈ 500 MB per iter × up to 100 iters.
- c-TF-IDF `chunk_tokens: Vec<Vec<String>>` materialises every token for every chunk simultaneously → 3–5 GB of transient String allocations.
- Projected peak: **6–10 GB steady-state + multi-GB allocator churn**, all anonymous heap.

**P2.** Four heavy cron jobs fire within 60 s of `DaemonPhase::Ready`:
- `src/cron/scheduler.rs:637–792`: git-history (10 s), similarity (30 s), graph (45 s), topic (60 s).
- All gated on `Ready`, which is set as soon as initial scan completes. For a warm index that's near-instant, so all four become overdue and serialise on the cron thread with no memory breathing room between them.

**P3.** 64-way ingestion of large JSONL files:
- `WorkPoolConfig.max_threads = 0` → `num_cpus::get()` = 64 workers.
- `src/indexer/processor.rs:47` does `std::fs::read_to_string(path)` unconditionally. `config.indexer.max_file_size_bytes` was consulted only *after* the read.
- Claude JSONL transcripts observed up to 43 MB each; 64 × 43 MB = **2.8 GB of file content in flight** during initial scan.
- WorkPool channels are unbounded by design (the adaptive hill-climber in `src/work_pool/monitor.rs` feeds `queue_depth` into its scaling objective — capping the channel would break the scaler), so queued closures themselves aren't the pressure point (~300 B each × 7500 files ≈ 2 MB). Active workers *holding content* are the pressure point.

**P4.** Graph analysis loads full per-project content:
- `src/cron/graph_analysis.rs:77–84` does `fetch_all` with content for every file in the project. For `claude` (7513 files, 50 KB avg) ≈ 300 MB per project.
- `insert_edges_batch` / `upsert_file_metrics_batch` were misnomers — they executed N sequential INSERTs per batch (for 500 rows: 500 round-trips), extending the wall-time window graph-analysis competed for RAM with topic-clustering.

| 8 | Mini-batch / online FCM for n > 1M | pending |
| 9 | Meta-clustering hierarchy on global topic centroids | pending |

Full plan at `/home/dylon/.claude/plans/when-i-launch-target-release-pgmcp-federated-porcupine.md`.

**Invariants preserved across all phases:**
1. Global full-chunk topic clustering (`scope = "global"`) remains the primary, always-on path — chunk-to-topic assignments must be cross-document-comparable.
2. Per-project clustering is an on-demand MCP tool AND a memory-pressure fallback — NOT a replacement for global.
3. WorkPool channels stay unbounded (adaptive hill-climber depends on queue depth as a scaling signal).
4. CPU FCM stays f32 (Zen 3 has no native fp16/bf16 arithmetic).

## 5. What did *not* work

- **Bounding the WorkPool channels**: initial hypothesis was that unbounded channels were allowing queue growth that caused memory pressure. Investigation of `src/work_pool/monitor.rs` revealed the hill-climber objective `J(N) = -w_tp·ema_tp + w_qd·ema_qd` uses `queue_depth` as a scaling signal. Bounding the channel would cap that signal and break the scaler. Queued closures are ~300 bytes each (Arc clones + PathBuf) — 7500 × 300 B ≈ 2 MB — so the queue itself was never the pressure point. Real P3 memory sink was *active workers holding file content*, addressed by the pre-read size gate.
  User correction recorded in memory: `feedback_adaptive_pool.md`.

- **Per-project clustering as a replacement for global**: initial plan considered this to eliminate the n×K-chunk FCM entirely. User corrected: global clustering is required for cross-document comparability of topic IDs. Per-project clustering produces topic IDs that are meaningless across projects. Plan adjusted to preserve global as primary, per-project as failsafe only.
  User correction recorded in memory: `feedback_global_fullchunk_clustering.md`.

- **f16 / bf16 on CPU**: Zen 3 has no native fp16/bf16 arithmetic. `cublasHgemm` and similar routed through software emulation via conversion to f32, giving no memory savings in the BLAS working set and adding conversion overhead. Rejected. fp16 on GPU (Ada Lovelace Tensor Cores) is the right path and is planned as Phase 5.

## 6. Reproducibility — exact commands

Measurement kit:
```sh
# Stable CPU frequency for benchmarks (per CLAUDE.md)
cpupower frequency-set -g performance

# Pin to specific cores while benchmarking
taskset -c 2-9 ./target/release/pgmcp <subcommand>

# /usr/bin/time peak RSS
/usr/bin/time -v ./target/release/pgmcp <subcommand> 2>&1 | tee /tmp/time-verbose.log
grep "Maximum resident set size" /tmp/time-verbose.log

# Constrained probe (reproduces OOM under hard cap to verify fix)
systemd-run --user --scope -p MemoryMax=4G -p MemorySwapMax=0 \
    ./target/release/pgmcp daemon

# Valgrind massif allocation profile
valgrind --tool=massif \
    --massif-out-file=/tmp/pgmcp-massif.out \
    --detailed-freq=10 --max-snapshots=200 \
    ./target/release/pgmcp <subcommand>
ms_print /tmp/pgmcp-massif.out > /tmp/pgmcp-massif-report.txt

# Continuous RSS sampling during daemon run
./target/release/pgmcp daemon &
PID=$!
while kill -0 $PID 2>/dev/null; do
    rss=$(awk '/VmRSS/ {print $2}' /proc/$PID/status 2>/dev/null)
    avail=$(awk '/MemAvailable/ {print $2}' /proc/meminfo)
    printf "[%s] RSS=%sMiB MemAvail=%sMiB\n" "$(date +%H:%M:%S)" \
           $((rss/1024)) $((avail/1024))
    sleep 5
done | tee /tmp/pgmcp-rss-sample.log
```

Prometheus gauges added in Phase 4 (live RSS telemetry without scraping /proc):
```
pgmcp_current_rss_bytes    — gauge, sampled every 500 ms
pgmcp_peak_rss_bytes       — gauge, fetch_max'd since startup
pgmcp_heavy_cron_running   — gauge, 1 iff a heavy cron body is executing
```
